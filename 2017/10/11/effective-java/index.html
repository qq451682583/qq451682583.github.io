  • 避免创建不必要的对象

  • 消除过期的对象引用

    内存泄露 ：

    • 如果一个栈先是增长，然后再收缩，那么从栈中弹出来的对象将不会被当做垃圾回收，即使使用栈的程序不在引用这些对象，他们也不会被回收，因为，栈内部维护着对这些对象的过期引用。这种情况可以在每次一个单元被弹出栈后，对此单元显示置空。

    • 注意：一般而言只要类是自己管理内存，程序员就应该警惕内存泄露问题问题。

    • 内存泄露的另一个常见来源是缓存 。一旦把对象的引用放到缓存中，它就很容易被遗忘掉，从而使得它不在有用之后很长一段时间内仍然留在缓存中。解决方案一般有几种，1.只要在缓存之外存在对某个项的键的引用，该项就有意义，那么就可以用WeakHashMap代表缓存；当缓存中的项过期之后，它们就会被自动删除。2.另外一种情形是，“缓存项的生命周期是否有意义”，这并不是很容易确定，随着时间的推移，其中的项会变得越来越没有价值。这种情况下，缓存应该时不时的清楚掉没有的项。。这项清楚工作可以由一个后台线程来完成，或者也可以再缓存中添加新的条目的时候顺便进行清理，LinkedHashMap类利用它的removeEldestEntry方法可以很容易的实现后一种方案。

    • 内存泄露的第三个常见来源是监听器和其它回调。注册回调之后，没有显式的取消注册，他们就会聚集。解决方案：确保回调立即被当作垃圾回收的最佳方法是只保存他们的弱引用。

  • 避免使用终结方法（finalize）

  • 覆盖equals时请遵守通用的约定

    1. 自反性。 对于任何非null的引用值X，x.equals(x)必须返回true
    2. 对称性。 对于任何非null的引用值x和y,当且仅当y.equals(x)返回true时，x.equals(y)必须返回true
    3. 传递性。 x,y,z x.equals(y)返回true y.equals(z)也返回true,那么x.equals(z)也必须返回true
    4. 一致性。 多次调用x.equals(y)就会一致的返回true，或者一致的返回false

    对于任何非null的引用值x，x.equals(null)必须返回false;

  • 覆盖equals时总要覆盖hashCode,相等的对象必须具有相等的散列码

  • 谨慎的覆盖clone
  • 考虑实现comparable接口

  • 使类和成员的可访问性最小化

    尽可能的使每个类或者成员不被外界访问

    长度非0的数组总是可变的 。类具有公有的静态final数组域，或者返回这种域的访问方法，这几乎总是错误的。

    例如：public static final Thing[] VALUES = {…};

    修正有两种方法：

    1 使公有数组变成私有的，并增加一个公有的不可变列表：

    pirvate static fina Thing[] PRIVATE_VALUE = {…};
    public static final List VALUSE = Collections.unmodifiableList(Arrays.asList(PRIVATE_VALUES));

    2 可以是数组变成私有，并添加一个公有方法，它返回私有数组的一个备份

    private static final Thing[] PRIVATE_VALUES = {…}
    public static final Thing[] values(){

    return PRIVATE_VALUES.clone();

    }

  • 复合优先于继承

    实现继承：当一个类扩展另一个类的时候
    接口继承：当一个类实现一个接口的时候，或者当一个接口扩展另一个接口的时候。

复合：不用扩展现有的类，而是在新的类中增加一个私有域，它引用现有类的一个实例。现有类变成了新类的一个组件，新类中的每个实例方法都可以调用被包含的现有实例中对应的方法并返回他的结果。这被称为转发，新类中的方法，被称为转发方法。
  • 要么为继承而设计，并提供文档说明，要么就禁止继承
  • 接口优于抽象类
  • 接口只用于定义类型，不应该被用来导出常量
  • 层次类优先于标签类
